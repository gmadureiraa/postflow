# Audit — fluxo do usuário pago · 2026-04-19

Percurso completo do cliente que **assina**, do signup até o reset mensal.
Status por etapa: ✅ funciona · ⚠️ existe mas tem gap · ❌ ausente/quebrado.

---

## 1. Landing → Login
**Status: ✅**

- `app/page.tsx` mostra hero + CTA pra `/app/login`.
- `/app/login` suporta 3 providers: Google OAuth, Twitter OAuth, email+senha.
- Analytics: PostHog `user_signed_in` / `user_signed_up`.

Gap: nenhum.

## 2. Trigger de criação de profile no signup
**Status: ✅**

- `supabase/schema.sql:160-175` define trigger `handle_new_user` que roda
  `AFTER INSERT ON auth.users` e cria row em `public.profiles` com
  `name`, `email`, `avatar_url` vindos do `user_metadata` do OAuth.
- RLS em `profiles` permite `INSERT` apenas pro próprio user — trigger roda
  como `SECURITY DEFINER` então bypassa RLS com segurança.

Gap: nenhum.

## 3. Onboarding guard (redirect /app → /app/onboarding)
**Status: ⚠️ tem gap pequeno**

- `app/app/layout.tsx:418-453` define `OnboardingGuard`. Lógica:
  - Se `loading`: retorna loader.
  - Se não autenticado: `router.replace("/app/login")`.
  - Se `profile && !profile.onboarding_completed && pathname !== "/app/onboarding" && pathname !== "/app/login"`: `router.replace("/app/onboarding")`.

Gap ⚠️: o guard é client-side (hook + useEffect). Há janela de **flash of
content** de ~50-200ms onde o dashboard renderiza antes do redirect. Não
é vulnerabilidade (RLS protege dados), mas é UX imperfeita.

Fix sugerido (não bloqueante): middleware que lê cookie do Supabase e
redireciona pra `/app/onboarding` se `profile_onboarding_completed` ainda
não estiver true. Complexidade: lê Supabase server-side no middleware
(~30min).

Para o admin: **⚠️ admin pode cair no guard de onboarding** se não
completou. Solução: admin com onboarding completo deve ser o padrão.

## 9. Webhook Stripe — pagamento confirmado
**Status: ✅** (com hardening pós-audit)

- Event `checkout.session.completed` (`app/api/stripe/webhook/route.ts:93-200`):
  - `profiles`: `plan = "pro" | "business"`, `usage_limit = 30 | 99999`,
    `usage_count = 0`, `stripe_customer_id`, `stripe_subscription_id`.
  - `payments INSERT` com `status = confirmed`, `plan`, `period_start/end`.
  - Email de confirmação via Resend.
  - Cupom marcado como redeemed.

- Gate novo (post security fix): em prod, se `STRIPE_WEBHOOK_SECRET` ausente,
  o endpoint retorna 503 — não processa eventos sem signature.

Gap: nenhum crítico. Falhas silenciosas de email não bloqueiam upgrade.

## 10. Uso mensal + quota
**Status: ✅**

- Free: 5 carrosséis/mês (`FREE_PLAN_USAGE_LIMIT=5`).
- Pro: 30/mês.
- Business: ilimitado (sentinel 99999).
- `/api/generate` rejeita com `403 PLAN_LIMIT_REACHED` quando
  `usage_count >= usage_limit` — frontend mostra toast + redirect pra
  `/app/plans`.

## 11. Falha de pagamento
**Status: ✅**

- Event `invoice.payment_failed`:
  - `payments INSERT` com `status = failed`.
  - Email ao user com link pro portal Stripe.
- Event `customer.subscription.deleted`:
  - `profiles.plan = "free"`, `usage_limit = 5`.

Gap ⚠️: email de falha usa Resend — se Resend cair, user não descobre que
caiu pra free até tentar gerar e levar 403. Mitigação já no código:
`profiles.plan` muda no webhook mesmo se email falhar.

## 12. Reset mensal
**Status: ✅**

- Cron `/api/cron/usage-reset/route.ts` roda dia 1 00:00 UTC.
- Chama RPC `reset_monthly_usage()` ou fallback com update direto.
- Zera `usage_count` pra todos (free, pro, business).
- Gate pós security fix: exige `Authorization: Bearer CRON_SECRET` (não
  mais bypass via `x-vercel-cron` header forjado).

## Resumo de gaps

| Gap | Severidade | Fix |
|---|---|---|
| Onboarding guard client-side (flash content) | Baixa | Middleware reading Supabase cookie |
| Email de falha pode não chegar | Baixa | Já mitigado — webhook atualiza plan mesmo assim |
| Admin não consegue acessar `/app/admin` se onboarding incompleto | Baixa | Admins já têm onboarding na prática |

**Conclusão**: o fluxo de assinatura está **pronto pra receber cliente
pagante**. Todos os passos críticos (auth → onboarding → uso → pagamento →
webhook → reset) funcionam. Gaps restantes são polimento de UX, não
bloqueadores.

## Verification manual sugerida

Antes de lançar em prod pra pagante real:

1. Criar conta teste com email `+test` do gmail.
2. Completar onboarding.
3. Criar 1 carrossel grátis → checar em `/app/admin/users/[id]` se aparece.
4. Stripe test card `4242 4242 4242 4242` → comprar Pro.
5. Webhook local via `stripe listen --forward-to localhost:3000/api/stripe/webhook`.
6. Ver `/app/admin?tab=subscriptions` — MRR + payment aparece.
7. Esperar 1 minuto, rodar `SELECT reset_monthly_usage()` no SQL Editor.
8. Verificar que usage_count zerou.
9. Cancelar sub no Stripe portal → webhook → user volta pra free.
